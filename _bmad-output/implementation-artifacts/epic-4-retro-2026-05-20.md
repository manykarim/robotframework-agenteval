## What worked substantively

### 1. The 4-reviewer cross-LLM pattern split into structurally-different reviewer lanes — and Auditor citation-drift is now its own near-certain band.

Pre-Epic-4, `feedback_n_way_agreement_weight` treated single-reviewer flags uniformly as "verify before applying." Epic 4 evidence demands a refinement: **Auditor 1-way HIGHs on PRD/ADR/architecture re-derivation are now near-certain when verified, distinct from generic single-reviewer flags**.

Track record across 5 epics (1a + 1b + 2 + 3 + 4): 6+ consecutive true-positives where Auditor solo-caught a citation/spec drift by re-deriving from the ratified source. Story 4.4 ratified this twice in one review (HIGH-B PRD FR10a L1499 `summary` nesting drift + HIGH-C Phase-1.5 carry-over catalog gap). Both 1-way, both production-correctness wins.

Why Auditor-1-way is special: the Auditor's methodology — opening the cited source + re-deriving — is structurally different from probe-based hunting. When Auditor solo-flags a citation drift, the other reviewers didn't see it because their hunting style doesn't include source re-derivation. The Auditor's 1-way is the *expected* result of correctly applying its methodology, not a confidence weakness.

Ratified as memory feedback: [[feedback_n_way_agreement_weight]] (extended).

### 2. `feedback_interleaved_dogfood_load_bearing` (Epic 3 retro) triply validated.

Stories 4.1, 4.2, 4.3 each had 3-way HIGHs that were INVISIBLE to fixture-only test coverage. Story 4.2's prompt-never-fed-to-stdin is the canonical example: 31 pre-edit tests all bypassed `_spawn`/`run()`; only the real-binary probe in code-review caught the adapter's production non-functionality.

The Epic 3 retro thesis — "future epics ship dogfood story alongside API surface" — wasn't perfectly executed in Epic 4 (no dedicated dogfood story shipped), but the interleaved-dogfood discipline showed up in the review prompts themselves. Each Story 4.x reviewer was asked to run behavioral probes; Codex's empirical execution was load-bearing every story.

Epic 5 should restore the explicit dogfood-story pattern (Story 5.5 explicit trace-assertion dogfood port already backlogged).

### 3. `feedback_test_name_assertion_match` (Epic 3 retro) caught fake-greens 2× more.

Story 4.3 review: 1-way HIGH on fake-green precedence tests where `assert len(results) == 1` would pass regardless of which adapter actually saw the model kwarg. Rewritten with `GenericAdapter.__init__` monkeypatch capturing actual kwarg flow.

Story 4.4 review: 2-way LOW-A (Blind + Codex) on Wilson CI bracket test where `assert t2.wilson_ci_lower == 0.0` (0/3) and `assert t3.wilson_ci_upper == 1.0` (3/3) verified only edge cases; the 2/3 mid-case bracketing — exactly what the test name promised — wasn't asserted. Rewritten with full bracketing invariant + canonical 2/3 at 95% reference values (0.208, 0.939).

Total catches across Epics 3 + 4: **6 fake-greens caught**. The norm is now load-bearing across 4 consecutive stories since Epic 3 retro ratification.

## What didn't, and what we learned

### 1. Carry-over catalog discipline lapse — caught a story later by Auditor cross-check.

Story 4.3 shipped 3 new Phase-1.5 carry-overs in spec + sprint-status (DF-4.3-S6 `@guarded_fanout` on `OrchestrationLibrary`; DF-4.3-S7 `resolve_config_with_provenance` refactor; DF-4.3-S8 dead `mcp_servers` filter closure) but **never updated `docs/phase-1-5-carry-overs.md`**. The gap surfaced a full story later when Story 4.4 Auditor cross-checked the catalog and raised HIGH-C.

Cost: one story of latency between the deferral commit and the catalog-of-record reflecting it. Risk: deferred items could be forgotten before Phase-1.5 work begins if the catalog stays out of sync.

The dev workflow had clean docstrings + clean spec; it simply never thought to update the catalog. A prose convention won't be remembered every time.

**Ratified as memory feedback: [[feedback_carry_over_catalog_gate]]** — story-close grep gate that verifies each new `DF-X-SY` pattern is in the catalog before flipping sprint-status to `done`. Pairs with [[feedback_spec_vs_ratified_doc_precheck]] as the *post*-story mirror.

### 2. No dedicated dogfood story in Epic 4 — but pattern survived via review-prompt discipline.

Per [[feedback_interleaved_dogfood_load_bearing]] (Epic 3 retro), "future epics ship dogfood story alongside API surface." Epic 4 shipped 4 API-surface stories but no dedicated dogfood story. The interleaved-dogfood discipline appeared instead in the cross-LLM review prompts (Codex behavioral probes against real production paths every story).

This worked but isn't sustainable as a substitute for explicit dogfood. Story 4.2's prompt-never-fed-to-stdin was caught by review-time real-binary probing, but the same bug in a less-reviewed area would have shipped.

Epic 5 backlog includes Story 5.5 (interleaved dogfood port rf-mcp trace assertions) — restore the explicit pattern.

### 3. Architectural gap: `MCPLibrary` excluded from `_SUB_LIBRARIES`, `@guarded_fanout` decorator can't reach it.

`MCP.Get Tool Discoverability` accepts `max_cost_usd` + `max_runtime_seconds` but can't enforce them because the `@guarded_fanout` decorator needs `_max_cost_usd` + `_max_runtime_seconds` on the host instance. Story 4.3 hit the same gap for `OrchestrationLibrary` initially (closed via `AgentEval._build_components` forwarding); MCPLibrary is structurally excluded per the Story 2.2 collision norm and constructed via `Library MCP WITH NAME MCP` rather than the AgentEval lifecycle.

DF-4.4-S1 (parallel to DF-4.3-S6) tracks the architectural fix for Phase-1.5. Tier-3 docstring badge surfaces the contract; operators must enforce budgets manually until cross-library config plumbing lands.

The gap isn't a Story 4.4 bug — it's an architectural carve-over from Story 2.2's collision-resolution decision. Worth surfacing for Epic 5 / Phase-1.5 planning.

### 4. DF-4.1-S2 + DF-4.2-S1 — adapter-side `mcp_servers=` integration deferred.

Both Generic adapter and Claude Code CLI adapter raise `NotImplementedError` on non-empty `mcp_servers`. Story 4.4's discoverability keyword therefore can't actually route tool dispatch via MCP — it dispatches WITHOUT MCP context. Meaningful real-LLM discoverability requires those carry-overs to land.

Epic 5 Story 5.2 (hosted-MCP observer) likely won't land this directly. The integration needs DF-4.1-S2 / DF-4.2-S1 closure first — possibly Phase-1.5 work. **Story 5.5 dogfood port assumes MCP-tool-call traces are emitted by the adapters, which currently they aren't.**

This is a real Epic 5 prerequisite that may need explicit decision: defer Story 5.5 until DF-4.1-S2 lands, or scope Story 5.5 to use the static-inspection MCP path only.

### 5. Pre-existing mypy errors in tests/ aren't gated.

102 mypy errors across tests/ at Epic 4 close (test_base.py, test_loader_smoke.py, test_trace_store.py, etc.). Not introduced by Epic 4 work — accumulated tech debt. Tests aren't gated by mypy currently; only src/ is.

Phase-1.5 hygiene candidate. Not Epic 5 blocking.
