## What worked substantively

### 1. The 4-reviewer cross-LLM pattern is now load-bearing in a different way than Epic 0.

Story 3.2 → Story 3.3 chain is the canonical evidence: Codex Probe 7 in Story 3.2 review surfaced `McpError("Connection closed")` widening from a synthetic crash. Story 3.3's actual rf-mcp port hit the SAME error class in production from a different cause path (missing `cwd=` → subprocess immediate-exit → SDK wraps as McpError). The widening was validated end-to-end in a different invocation context. That's the cross-LLM pattern catching bugs that single-family review misses + the next story validating the catch in production.

### 2. Interleaved-dogfood thesis VALIDATED (AC-DOGFOOD-01).

DOGFOOD-FINDING-1 (RF↔SDK errlog `fileno` crash) escaped Story 3.1's 4-reviewer cross-LLM code review because unit + RF integration tests used the in_memory transport. Only a real downstream subprocess server (rf-mcp's robotmcp) exposed it. Pre-fix: 11 of 15 parity tests failed. Post-fix: 15/15 pass. **The interleaved-dogfood story design — Story 3.3 shipping in the same epic as the API-surface stories — paid for itself in week-3 instead of week-10.**

Ratified as memory feedback: `feedback_interleaved_dogfood_load_bearing`.

### 3. Per-call-session pattern (agentguard-inspired) survived 3 stories of pressure.

Story 3.1 ratified it. Story 3.2 factored shared helpers (`_validate_handle_for_connect`, `_open_session`, `_initialize_with_typed_error_mapping`). Story 3.3 ran it against a heavy real server. No mid-epic refactor required — sign that the design absorbed reality. Phase-1.5 may introduce pooled sessions (DF-3.3-S5/S7 anticipates this) but Phase-1 ships honestly with the simpler shape.

### 4. Pre-create-story drift check escalated to 17 uses; caught 11 drifts pre-authoring across the epic.

Story 3.2: 5 drifts (3 HIGH + 2 MED) including 17th leaf MCPConnectionLostError catalog amendment.
Story 3.3: 6 drifts (3 HIGH + 2 MED + 1 LOW) including the scope-narrowing decision that vendoring in agenteval's `tests/dogfood/` is correct vs vendoring in rf-mcp itself (Story 9.1 territory).

Without the precheck, those drifts would have landed in commits + been caught downstream — expensive.

## What didn't, and what we learned

### 1. Fake-green test pattern hit us TWICE.

- Story 3.2 Blind HIGH-2: `test_call_tool_arguments_are_copied_not_referenced` asserted nothing meaningful — the assertion would pass regardless of whether `call_tool` did the defensive copy. Removing the copy would have left the pre-edit test green.
- Story 3.3 Blind HIGH-2 + Edge-cases H3: `Robotmcp Execute Step With Invalid Keyword Yields Is Error` only asserted `${result.latency_ms} > 0` (passes for ANY successful round-trip). The intended disjunction lived in a code comment, not an assertion.

**Both caught by Blind reviewers reading the test name first + the body second.**

Ratified as memory feedback: `feedback_test_name_assertion_match`. The cross-LLM review prompt extends: "check every test's assertion body delivers on the test name's claim".

### 2. Pre-create-story drift check is necessary but NOT sufficient.

Two distinct Epic 3 cases:
- Story 3.1 Auditor HIGH: PRD FR8 said `"server version <X>..."`, shipped impl had `"MCP "` prefix. PRD wording stale relative to impl.
- Story 3.2 Auditor HIGH: PRD FR9b said `(result, error, latency_ms)` 3-field shape; impl + epics.md L1271 had 5-field shape (`content, is_error, error_message, latency_ms, correlation_id`).

Both escaped pre-create-story drift check because the check scopes to known doc-pairs (ADR↔epics, epics↔story). The Auditor reviewer caught both post-implementation via re-derive-from-source.

`feedback_citation_drift_first_class` EXTENDED with Epic 3 ratification:

1. citation-drift-first-class auditor remains the **load-bearing safety net**; pre-create-story drift check is the early-detection layer, not the only layer.
2. Auditor "re-derive every cited fact from source" instruction must be unconditional — even when the spec author claims pre-create-story drift check ran clean.
3. Future norm refinement (Phase-1.5): widen pre-create-story check to compare story-spec FIELD SETS against PRD FR-body wording.

### 3. Codex CLI sandbox process gap (DF-3.2-S7 + DF-3.3 recurrence).

The bwrap loopback denial blocked `git show` / `cat` / `rg` in the codex subprocess. Story 3.2's review filled the slot with behavioral probes. Story 3.3's Codex recovered via web-search fallback. Both reviews remained load-bearing — the slot's value is the **independent-reasoning-LLM-family invariant + the behavioral-probe discipline**, not literally Codex-the-binary. But this is a real process gap; the cross-LLM-family norm is degraded until we either fix the sandbox or adopt a sibling CLI (Gemini, Mistral, etc).

Tracked as DF-3.2-S7. Many opted NOT to ratify a new memory feedback for this (the existing `feedback_review_methodology_norms` already covers the parent norm). Phase-1.5 priority.

### 4. Fake-green AC-MCP-OBSERVE-02 on primary production transport (Story 3.1).

The Story 3.1 version-gate test exercised our `version_gate.py` + monkeypatched `mcp.shared.version.SUPPORTED_PROTOCOL_VERSIONS`. But `mcp.client.session` module-load-snapshots that symbol — so the SDK saw the original allowlist + the test was green for the wrong reason on the **primary production transport (stdio)**. Edge-cases + Codex 2-way HIGH caught it; load-bearing refactor in `transport.py` (return UN-INITIALIZED sessions; lifecycle.py runs initialize() under agenteval control + maps SDK's bare RuntimeError to typed error).

Lesson: test coverage of code we wrote ourselves is not the same as test coverage of code-under-test. Behavioral probes against real SDK invariants are the resolution. This is part of why `feedback_codex_probe_fitness` was ratified at Epic 2 retro.
