## What didn't, and what we learned

### 1. Cross-LLM critical review pipeline degraded sharply — single-reviewer (or zero-reviewer) for 4 of 5 stories

**This is the load-bearing finding of Epic 8 retro.** Only Story 8a.1 received a substantive cross-LLM review (Claude caught HIGH-1 `cost_usd` key mismatch — production silent bug). Stories 8a.2, 8b.1, 8b.2, 8b.3 all degraded to single-reviewer (or zero-reviewer) because:

- **Codex CLI rate-limited from Story 8a.1 onward.** OpenAI's "You've hit your usage limit. Upgrade to Pro... try again at 3:38 PM" message blocked all subsequent invocations. 3+ retries across the epic all failed.
- **Claude CLI returned empty output on 4 of 5 long prompts** (Stories 8a.2 + 8b.1 + 8b.2 + 8b.3 review prompts). The `claude --print --output-format text "$(cat /tmp/prompt.md)"` invocation either timed out silently (8+ min, exit 0, 0-byte output) or hit some other failure mode not visible to the operator.

**Consequence:** the project's ratified cross-LLM review norm (`feedback_review_methodology_norms`) was effectively suspended for 4 of 5 stories. We mitigated via:

- End-to-end integration tests (Section 1 "What worked").
- Empirical probes (DebugListener pattern in Story 8a.2).
- Self-review with explicit drift-check + caller-count + nullish-fuzz checklists.

**The mitigation worked** — every empirical bug was caught before commit — but the project lost the cross-LLM signal that has been load-bearing for 30+ consecutive STAR catches across Epics 2-7. **This is a process risk that must be addressed before Epic 9.**

**Lesson:** the cross-LLM review pipeline needs (a) a fallback CLI invocation pattern that works reliably (per `/goal` directive: `claude -p --dangerously-skip-permissions --model opus "Query"` — the `-p` short form + explicit `--model opus` may have different stability characteristics than `--print --output-format text`), (b) a quota-budget tracker for Codex to avoid silent rate-limit fail, and (c) a third LLM family (Gemini / Mistral CLI) to restore the two-family invariant when one is unavailable. **This is Action #1 for Epic 9 retro.**

### 2. The listener resolution issue (Story 8a.2 D-6) reveals a 5-story-old contract drift

`docs/contracts/listener-integration.md` L38 documented `--listener AgentEval.telemetry.listener` as canonical since Story 5.1 (ratified 2026-05-19). Story 8a.1's smoke test used the short form + "appeared to work" because the no-op `xunit_file` had no observable side effect. Story 8a.2 was the first story with a directly-observable listener-hook side effect (the `trace_id` tag in `output.xml`) — which surfaced that the short form silently fails on RF 7.x. **5 stories shipped using a broken canonical invocation that worked only because their listener hooks happened to be no-ops.**

The blast radius: any user following `listener-integration.md` since Story 5.1 close had been running with the listener silently disabled. Story 8a.2's fix amended 3 contract docs + the integration test; Story 8b.1 propagated the fix to all scaffolded files + Recipe #1; Story 8b.3 propagated to all 8 recipes + Recipe #4 polish.

**Lesson:** the contract `listener-integration.md` was authored in Story 5.1 + never empirically smoke-tested against a real RF subprocess. The Story 5.1 unit test (`test_listener_entry_point_resolves_to_listener_class`) verifies the **entry-point** resolution, which is a DIFFERENT mechanism from RF's `--listener` flag resolution. **Candidate new norm: contract documents that document CLI invocations MUST carry a subprocess-based integration smoke test verifying the documented invocation works end-to-end.** This is broader than `feedback_executable_doc_precheck` (which covers doc code blocks); it covers contract-doc invocations specifically.

### 3. Listener tag-add API surface (Story 8a.2 D-5) — `data.tags` vs `result.tags` empirical disambiguation

RF Listener v3 docs are ambiguous on which `tags` object surfaces in `output.xml`. Empirical truth (verified via DebugListener probe 2026-05-25): only `result.tags.add(...)` lands in `output.xml`; `data.tags.add(...)` is a no-op. **The project's listener.py docstring at Story 5.1 didn't document this empirically-verified contract.** Story 8a.2's listener.py now carries the disambiguation; the `listener-integration.md` contract should also be amended.

**Lesson:** when extending a Listener v3 hook to write data that surfaces in `output.xml`, smoke-test BOTH `data.tags` and `result.tags` empirically before committing. The **DebugListener probe is committed as a reusable artifact** at `tests/integration/probes/debug_listener.py` (committed 2026-05-25 per cross-LLM review MED-2 [Claude]: the inline-snippet form has no runnable foothold; a Phase-2 dev re-deriving this norm needs a concrete entry point).

Canonical invocation (full instructions in the probe's module docstring):

```bash
uv run robot --listener tests/integration/probes/debug_listener.py \
    --output /tmp/probe-output.xml --report NONE --log NONE /tmp/probe-suite.robot
grep "from_data\|from_result" /tmp/probe-output.xml
# Empirical truth on RF 7.x: only `<tag>from_result</tag>` surfaces.
```

**Candidate amendment to `listener-integration.md`:** add a "Listener-Hook API Surface" subsection documenting which mutations surface in `output.xml` vs which are no-ops.

### 4. Recipe CI extraction (DF-8b.3-S1 / C64) is the largest deferred-work item from Epic 8

The story spec called for "all 8 recipes pass as executable examples (each recipe's code block is extracted by CI + run as a smoke test)" (`_bmad-output/planning-artifacts/epics.md` L1931). **Lesson:** the `feedback_executable_doc_precheck` norm (Epic 7 retro NEW) covers code blocks in shipped docs but the *infrastructure* to enforce it across 8 recipes is non-trivial. Story 8b.3 D-3 made the path-of-least-amendment decision (Recipe #2 only); the risk is real. **Action #2 for Epic 9 retro: prioritize C64.**

## Norms ratified this retro

1. **NEW: `feedback_listener_hook_api_surface_empirical_check`** — when extending an RF Listener v3 hook to write data that surfaces in `output.xml`, empirically verify which API surface (`data.tags` vs `result.tags`, etc.) is the production-correct one via the canonical DebugListener probe committed at `tests/integration/probes/debug_listener.py`.
