## Deferred from: code review of 0-1-run-hosted-mcp-universal-observer-spike (2026-05-17)

The 20 items below are real defects in the Story 0.1 spike's scratch Python code. They do **not** block the spike's verdict (`AMEND-ADR-007`) because the spike's deliverable is the findings document, not production code. They DO block the production observer that Epic 5 Story 5.2 will build.

### Carry-overs to Epic 5 Story 5.2 (`src/AgentEval/mcp/observer.py`)

- `attach()` idempotency: double-attach must not double-count tool calls; raise or no-op on re-attach to the same server.
- SDK exception swallowing: production observer must inspect `ServerResult.isError` to distinguish tool failures from successes (the SDK absorbs tool exceptions into `_make_error_result`; the observer's `try/except` around `await handler(req)` is dead for normal tool errors).
- Encapsulation: no external code may mutate `_seq` / `_traces` / private state. All trace ingestion goes through a single `_record()` method. The spike's `_graft_subprocess_log` pattern (free-function reaching into observer internals) must NOT survive into production.
- `mark_external_mixed` must accumulate reasons: `list[str]`, not single overwrite. Declare the field in `__init__`, not as an ad-hoc instance attribute.
- `_summarize_result` must handle multi-block / non-text-first / `ImageContent`+`TextContent` mixed results — not just `blocks[0].text`.
- Arguments serialization: validate JSON-serializability at capture time, not defer to `write_jsonl`; consider Pydantic models for the trace shape.
- Trace immutability: deep-copy args at record time, OR document that callers must not mutate args after a tool call.
- `mcp_coverage` decision logic + `observed_paths` ordering: implementation must mirror the ratified ADR-A6 semantic (whichever D1 resolves to); `observed_paths` should reflect decision order, not alphabetical.
- `finalize()` terminal-state guard: subsequent `finalize()` calls return the same `run_id` + frozen trace snapshot, OR raise.
- Subprocess-side observation: production `subprocess_with_observer` path must inject the observer at subprocess bootstrap (the actual handler-wrap pattern), NOT rely on cooperating servers that log to JSONL. See D2 in story review findings.
- Performance: subprocess-side `_append_log` should batch writes (e.g., write at handler exit, not per-call open/close) for high-frequency tool runs.
- Locking: document single-producer-per-observer assumption OR add anyio.Lock for the `_seq` increment if multi-concurrent-clients can hit the same observer.

## Dogfood findings from rf-mcp MCP-surface port (Story 3.3, 2026-05-19)

Surfaced by porting a representative subset of rf-mcp's pytest MCP-surface tests (1128 LoC across 4 files) to `.robot` using agenteval Epic 2 + Epic 3 keywords. 15-test parity suite at `tests/dogfood/rf-mcp/test_mcp_surface_parity.robot` now passes 15/15 against the live robotmcp server. AC-3.3.8 ("≥1 actionable improvement filed") — surfaced 2 REAL findings, not invented to satisfy the AC.

- **DOGFOOD-FINDING-1 (DF-3.3-FINDING-1, HIGH, FIXED IN STORY 3.3)** — `io.UnsupportedOperation: fileno` under Robot Framework runtime. The MCP SDK's `stdio_client(server, errlog=sys.stderr)` default crashes under `robot` execution because RF listener replaces `sys.stderr` with a non-fd capture buffer + the SDK calls `.fileno()` on the errlog. Pre-fix: 11 of 15 parity tests failed (every stdio-using test). Fix: pass `errlog=sys.__stderr__` explicitly at `src/AgentEval/mcp/transport.py:open_stdio_session`. The pre-Story-3.3 comment "errlog is NOT passed to avoid this" was misleading — the SDK default WAS `sys.stderr`, so the comment was wrong AND the fix was missing. Classic interleaved-dogfood catch: bug existed since Story 3.1 + escaped Story 3.1's 4-reviewer cross-LLM code review because the unit tests + RF integration tests used the in_memory transport (which doesn't go through `stdio_client`). The stdio path needed a REAL downstream subprocess server (rf-mcp's robotmcp) to expose it.

- **DF-3.3-S1 (DOGFOOD-FINDING-A, MED, WORKAROUND APPLIED IN STORY 3.3, REAL FIX PHASE-1.5)** — `MCP.Start Server` + `MCPServerHandle` + `start_server` function lack a `cwd=` parameter. rf-mcp's `.mcp.json` declares `command="uv", args=["run", "-m", "robotmcp.server"]` — `uv run` resolves the project from cwd, but agenteval has no way to set subprocess cwd. Workaround (Story 3.3): inject `--directory ${RF_MCP_REPO_ROOT}` into the args list at .robot suite setup. Real fix: add `cwd: str | None = None` field to `MCPServerHandle`; thread through `start_server` + `_open_session` + `open_stdio_session` + `StdioServerParameters(cwd=...)` (SDK already supports cwd). Additive; non-breaking. Effort S. Phase-1.5 hygiene.

- **DF-3.3-S2 (DOGFOOD-FINDING-Story 3.2 VALIDATION, INFO)** — Story 3.2's `McpError("Connection closed")` → `MCPConnectionLostError` widening (Codex HIGH-2 Probe 7) was VALIDATED IN PRODUCTION by Story 3.3's dogfood port. When the robotmcp subprocess immediately exited (before DF-3.3-S1 workaround) with `ModuleNotFoundError`, the SDK raised `McpError("Connection closed")` — and the Story 3.2 widening correctly mapped it to the typed `MCPConnectionLostError` with `server_name`/`last_operation`/`fix_suggestion` attrs. Pre-Story-3.2 the operator would have seen a raw `McpError` with no structured diagnostics. The 18th consecutive cross-LLM STAR catch is now load-bearing in real dogfood. INFO-only carry-over (no action needed); kept as evidence for future code-review retros.

- **DF-3.3-S3 (DOGFOOD-FINDING-config-key-typo, LOW, DEFER)** — `MCP.Get Server Config` parser tolerates unknown top-level JSON keys (e.g., `_vendoring_note`). This is benevolent for vendoring annotations but ALSO means a typo like `mcpServer` (missing trailing `s`) silently returns an empty dict instead of raising `InvalidMCPServerConfigError`. Risk: a user-authored `.mcp.json` with the typo passes static inspection silently. Phase-1.5: tighten parser to raise on unknown top-level keys OR add an explicit `mcpServers`-presence assertion to `parse_mcp_servers`. Effort XS. Catalog amendment of `InvalidMCPServerConfigError` docstring to enumerate the new raise condition.

- **DF-3.3-S4 (DOGFOOD-FINDING-CI-workflow-gap, MED, deferred to Story 9.1 per spec D-B)** — `dogfood-integration.yml` (Story 1a.2 install-smoke-only) does NOT clone rf-mcp + run the parity suite on PR. Phase-1.5 hygiene: track the workflow extension explicitly in Story 9.1 spec when it's authored.

- **DF-3.3-S5 (DOGFOOD-FINDING-pabot-not-verified, LOW; PARTIALLY VERIFIED IN STORY 3.3 CODE-REVIEW)** — AC-3.3.9 requires `pabot --processes 4` parallel execution with no orphan-process leak. Phase-1.5 hygiene: pin the `--testlevelsplit` requirement in the suite header OR refactor pabot invocation pattern to be parallelism-by-default.

- **DF-3.3-S6 (FROM-STORY-3.3-CODE-REVIEW, Codex MED-2)** — stderr-interleave under parallel execution. `errlog=sys.__stderr__` is the process-global fd 2; N parallel pabot workers all multiplex their child robotmcp stderr onto one fd. - **DF-3.3-S7 (FROM-STORY-3.3-CODE-REVIEW, Blind MED-3)** — SIGKILL / event-loop-panic subprocess leak path. Because each `Call Tool` spawns + reaps its own subprocess (Phase-1 per-call-session design), the per-call `try/except BaseException → stack.aclose()` at `transport.py:144-146` is the only reaper. - **DF-3.3-S8 (FROM-STORY-3.3-CODE-REVIEW, Blind MED-4 + Edge-cases L2)** — `--directory` workaround is fragile against `uv run` flag ordering. The Story 3.3 .robot suite injects `['--directory', $RF_MCP_REPO_ROOT]` to work around the missing `cwd=` parameter. - **DF-3.3-S9 (FROM-STORY-3.3-CODE-REVIEW, Edge-cases M4)** — `ROBOTMCP_ATTACH_PORT=7317` collision under parallel pabot. rf-mcp's `.mcp.json` pins port 7317 across all parity-suite invocations.
