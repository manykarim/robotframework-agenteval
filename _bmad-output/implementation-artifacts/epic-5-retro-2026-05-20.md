## What didn't, and what we learned

### 1. In-flight spec amendments during dev are NOT caught by `feedback_spec_vs_ratified_doc_precheck`

The pre-create-story drift check fires BEFORE story authoring. But Stories 5.3, 5.4, 5.5 ALL had spec amendments DURING dev that weren't propagated to all artifacts in lockstep:

- **Story 5.3**: HIGH-E (FR39 path `run-manifest__` → `manifest__`) caught by Auditor at code-review, not pre-create-story.
- **Story 5.4**: HIGH-G (FR34b verbatim box-drawing visual format) — spec ratified ASCII `===` but FR34b mandated `┌─ EVIDENCE ─┐`. Drift caught by Auditor at code-review.
- **Story 5.5**: HIGH-G + HIGH-H (AC-5.5.3 #4 silently dropped + #1 substantively replaced). Spec AC text was NOT amended when dev decisions changed the shipped tests.

The pattern: dev makes a reasonable adaptation (different test, different field name, different exception) but doesn't amend the AC text in lockstep. The spec drifts silently. Auditor catches at code-review.

**Lesson:** the cross-LLM Auditor is currently load-bearing for catching in-flight spec drift. This works (11+ consecutive TPs) but it's a late-binding catch. A dev-side norm could move the catch earlier.

### 2. Dogfood tests have a HIGHER fake-green base rate than regular tests

Story 3.3 dogfood code-review caught 2 fake-greens at HIGH severity. Story 5.5 dogfood code-review caught **4 fake-greens** at HIGH severity (HIGH-D × 2 + HIGH-C + HIGH-E). That's a 200% jump in fake-green findings per dogfood story.

Possible causes:
- Dogfood tests stretch toward "real" assertions but Phase-1 infrastructure gaps (no agent loop, no adapter span emit) make full-fidelity assertions impossible.
- Dogfood test authors (me, Amelia) reach for `Should Be Equal ${value} 0` empty-list assertions because they're easy, without checking whether the empty-list outcome distinguishes the implementation contract from environmental no-op.
- `[Tags] slow` exclusion from pytest means dogfood failures don't surface until manual `uv run robot` — reduces the "did the test catch a bug?" feedback loop.

**Lesson:** dogfood tests need a stricter pre-write checklist. The `feedback_test_name_assertion_match` norm fires at code-review — but for dogfood it should fire at story-write time, before the .robot file is even committed.

### 3. Adapter span instrumentation gap (DF-5.5-DOGFOOD-2) was latent across all of Epic 5

`telemetry/spans.py` ships `invoke_agent_span` + `chat_span` + `execute_tool_span` context-manager helpers — but NO adapter (`GenericAdapter`, `ClaudeCodeCLIAdapter`) or provider calls them in `run()` / `chat()`. As a result:

- `Get Spans test_id=<test>` returns `[]` for any Phase-1 test that doesn't manually wrap work.
- The Story 5.5 dogfood test #1 (`Trace Store Records Spans When Robotmcp Tool Called`) was unimplementable as originally specified — substituted by the vendored-config sanity test (HIGH-H finding).
- Epic 6 Story 6.1 (`Metric.Get Tool Call Count` etc.) WILL hit this gap immediately — metrics read from spans.

The gap existed since Story 5.1 shipped span helpers but Story 4.1/4.2 adapters didn't wire them. It surfaced only at Story 5.5 dogfood time when an integration probe needed to assert on `Get Spans`. **5 stories of latency between gap creation and detection.**

**Lesson:** at Story 5.1 close (the story that introduced the span helpers), the dev-close checklist could have included "verify caller count > 0 for new public helpers via `grep -rn`. If zero callers, flag as DF-X-SY caller-gap entry." Would have surfaced the gap at Story 5.1 close, not Story 5.5.

### 4. The `__suite__` sentinel design (Story 5.4) was incomplete — Story 5.5 dogfood proved the gap

Story 5.4 introduced `_PRE_TEST_BUFFER_KEY = "__suite__"` as a routing sentinel for `record_warning` calls without a bound Listener context. The implementation handled "merge sentinel into first bound test" via `flush_pre_test_buffer(test_id)`. But:

- The sentinel was NOT exposed via the public `Get Last Warnings` keyword surface.
- `Get Last Warnings test_id="current"` returned `[]` from Listener-less contexts.
- `Get Last Warnings test_id="all"` excluded the sentinel per the AC-5.4.5 design.
- Listener-less consumers (the Story 5.5 dogfood) couldn't reach their own emitted records.

The gap was filed as DF-5.5-DOGFOOD-4 → C45 → CLOSED same-PR. But conceptually, the sentinel design's "Listener-less context unreachable" property should have been caught at Story 5.4 code-review time, not at Story 5.5 dogfood time.

**Lesson:** when a story introduces a routing sentinel / fallback bucket for state, the public-surface accessor should expose the bucket explicitly, OR the spec should document the unreachability + provide an escape hatch. Story 5.4 did neither.

### 5. AC-5.5.3 spec drifted with dev decisions — but the spec text wasn't amended in lockstep

In Story 5.5, dev decided:
- Original AC test #1 (`Trace Store Records Spans When Robotmcp Tool Called`) was unimplementable per DF-5.5-DOGFOOD-2 → substituted by `Rfmcp Vendored Config Parses Cleanly`.
- Original AC test #4 (`Run Manifest Sidecar Written To Output Directory`) required a full Listener cycle the dogfood couldn't drive → covered indirectly by test #8's full emit-pipeline + schema-validate.

Both decisions were reasonable. But the AC-5.5.3 spec text WASN'T amended to reflect them. The Auditor caught both at code-review as HIGH-G and HIGH-H. Spec was amended post-hoc.

**Lesson:** in Story 5.x reasonable substitutions deserve a same-commit AC text amendment. Add to the dev-close checklist: "did this story's tests match the AC text verbatim? If not, amend AC text in the same commit." Closes the loop before the Auditor has to catch it.
