## What worked substantively

### 1. The pre-create-story drift check is now load-bearing for usability gaps, not just citation drift

The original `feedback_spec_vs_ratified_doc_precheck` norm (Epic 1a ratification) targets **spec-vs-ratified-doc drift** — a citation-level check. But across Epic 7, the check fired on a different class of problem in each story:

- **Story 7.1 D-1**: architecture L1274 named `SkillActivationResult`; PRD-architecture lookup vs implementation plan revealed it should be `ActivationDecision` (rename pre-authoring).
- **Story 7.1 D-2**: `SkillsLibrary` excluded from `_SUB_LIBRARIES` → `@guarded_fanout()` budgets default to `None`; library-level `max_cost_usd=` does NOT thread through. Filed as DF-7.1-S1 / C55. **This is a usability gap, not citation drift.**
- **Story 7.2 D-2**: `mcp_coverage` (Story 1b.2 concept) vs `adapter_coverage` (the field this story actually needed) — naming distinction per ADR-016 §Scope. Caught + corrected pre-authoring. **Architectural concept disambiguation, not citation drift.**
- **Story 7.2 D-3**: Phase-1 heuristic cannot determine WHICH competing skill the agent chose — only whether the TARGET skill was mentioned. `competing_skills_picked` always `{}` in Phase-1. Catalogued C56 BEFORE the story shipped. **Heuristic limitation disclosure, not citation drift.**
- **Story 7.3 D-1 (HIGH)**: `Stat.Get Pass At K` default predicate (`completeness=="complete"`) returns 0.0 for `ActivationDecision` (no `metadata.completeness` attribute → silent zero). **Empirically verified pre-authoring** — caught before integration test was written; documented via DF-7.3-S1/C59 + custom predicate pattern threaded through test docstrings, recipe NOTE comment, and dev notes.
- **Story 7.4 D-1**: stub adapters always embed target skill name in response → activation heuristic fires for every trial → `false_activation_rate=1.0` by design. **Dogfood limitation disclosure** — would have surfaced as a fake-green at code-review without the precheck.

**The check is now operationally a usability + heuristic-limitation + concept-disambiguation gate, not just a citation gate.** The 100% catch rate (34 consecutive uses) is sustained because the scope of "drift" the prompt now interrogates has implicitly widened.

### 2. The cross-LLM 4-reviewer pattern caught a real Phase-1 heuristic bug on Story 7.1 (HIGH-1)

`get_activation_decision`'s activation heuristic is `fm["name"].lower() in result.response_text.lower()`. The unit tests covered: empty-string name → False; matching name → True; case-insensitive match → True.

**What was missed pre-review**: when YAML frontmatter contains `name: null` (or `name:` with no value), `fm.get("name")` returns `None`, and `str(None).lower() == "none"` — meaning ANY agent response containing "none" (as in "none of the above", "found none") spuriously activated. Caught by Claude Blind Hunter at HIGH-1; fixed via `isinstance(name_raw, str) else ''` guard + `test_null_skill_name_in_yaml_gives_activated_false` regression test.

This is the kind of edge case `feedback_codex_probe_fitness` predicts (type system saw a perfectly valid `str | None` projection) and `feedback_test_name_assertion_match` is supposed to catch (the dev's test list didn't probe the `None` YAML path). The cross-LLM review is the load-bearing safety net.

## What didn't, and what we learned

### 1. Story 7.3 recipe RF block shipped to dev-review with broken keyword syntax — caught only at cross-LLM review

The recipe stub authored during Story 7.3 dev contained:

```robotframework
Run N Times    n=10    keyword=Skill.Get Activation Decision    keyword_args=skill=X    prompt=Y    adapter=Z
${pass_at_k}=    Stat.Get Pass At K    runs=${runs}    k=5    predicate=${lambda r: r.result.activated}
```

Both lines are RF-broken: `keyword_args=skill=X    prompt=Y    adapter=Z` parses as 3 separate named args to `Run N Times` (which has no `prompt`/`adapter` params → runtime error). `${lambda r: ...}` is RF variable lookup syntax, not Python eval → `VariableNotFoundError` at runtime.

Caught by Blind Hunter + Codex CLI as a 2-way HIGH. Fixed via `Create Dictionary` for `keyword_args` + `${{lambda r: ...}}` double-brace inline eval.

**Lesson:** when a story produces a documentation artifact (recipe, README example, libdoc fragment), the dev's "all tests pass" verification does NOT cover whether the artifact actually executes as documented. The cross-LLM review is currently load-bearing for this. A dev-side norm could move the catch earlier: **executable-doc precheck — for every code block in shipped docs, run a minimal smoke-execute of the RF snippet before flipping to review.** Would have caught both broken lines at write time, not at code-review.

### 2. Story 7.1 ship-blocking null-name bug was unit-test-blind — the dev's test list didn't probe `None` from YAML

`test_activation_decision.py` shipped with 12 tests (later 14 after the regression patch). The dev's coverage list included:
- Empty-string name → False (good — but `""` is not `None`)
- Matching name → True
- Case-insensitive match → True
- Adapter cost+latency passthrough
- `polling=` raises `PollingDisallowedError`
- `@tier(3)` decoration
- `@guarded_fanout()` decoration
- `_split_adapter_kwargs` integration

The dev did NOT write a test for `name: null` or `name:` (empty value) in YAML frontmatter — even though the `parse_frontmatter` contract returns `dict` and YAML `null` keys are a standard idiom. The bug was caught by Blind Hunter HIGH-1 at code-review, then a regression test added.

**Lesson:** for string-coercion code paths where the source data comes from external untrusted YAML/JSON, the dev test plan MUST include the 5 nullish-value variants: `None`, `""`, `False`, `0`, missing-key. This is a known idiom but not currently a norm. **Candidate new norm: "nullish-input fuzz checklist for external-data parsers — 5 variants required at unit-test authoring time."**

### 3. Story 7.4 dogfood validates infrastructure correctness — not real activation quality

Story 7.4 ships 3 vendored skill files + 3 task YAMLs + 4 robot tests + 4/4 passing. But the stub adapters used (no live LLM call) ALWAYS embed the target skill name in the response → activation heuristic fires for every trial → `false_activation_rate=1.0` for all decoy tasks **by design**.

The story's spec acknowledged this (D-1 pre-create) and catalogued DF-7.4-S1/C60 → Epic 9. The dogfood verifies:
- Task YAML parsing
- Aggregation logic (per-task + summary)
- Robot Framework keyword wiring (Library import path; argument shapes)
- Numeric assertions match observed stub-result values (no fake-green at the assertion level)

But it does NOT verify:
- Real false-activation discrimination
- Real missed-activation discrimination
- Realistic prompt-to-skill match quality

**Lesson:** dogfood with stub adapters has a structural ceiling. The "dogfood-finding severity-differentiation" norm (Story 5.5 retro #4) correctly classifies this as Epic 9 work — but the framing in the dogfood docstring should be MORE explicit. A `WARNING: stub-only — activation quality is decorative until Epic 9 live-provider run` banner at the top of `parity-checklist-agentskills-discoverability.md` would prevent future readers from over-interpreting the 4/4 pass. **Candidate clarification: dogfood parity-checklist files MUST carry a top-of-file "VALIDATION-CEILING" line stating what the dogfood does and does not verify.**

### 4. The "in-flight spec amendment" norm (Epic 5 retro #1) fired late on Story 7.3 AC-7.3.4

Story 7.3 AC-7.3.4 originally specified the Pass@k predicate as `lambda r: r.result.activated`. The dev changed it during implementation to `isinstance(r.result, ActivationDecision) and r.result.activated` (defensive isinstance guard — because the predicate runs on heterogeneous `KeywordRun` results when `Stat.Run N Times` is misconfigured). The AC text was NOT amended in lockstep — caught by Acceptance Auditor at code-review (MED), then amended.

This is the 4th instance of the in-flight-spec-amendment norm firing at code-review time across Stories 5.3, 5.4, 5.5, 7.3. The norm exists; the dev-side checklist already includes it; the catch still happens at code-review.

**Possible cause:** the dev-side checklist runs at story-close (Task N-1 / N) but the amendment happens at code-write time (mid-task). The checklist asks "do the shipped tests match the AC text?" — by the time the answer is checked, the diff has been forgotten.

**Lesson:** the in-flight-amendment check should fire AT the moment the dev edits an AC-derived line, not at story-close. Currently the norm is "verify at close-out"; a stronger variant would be "every diff hunk touching an `AC-X.Y.Z`-derived line MUST also touch the AC text." This is hard to automate without a git pre-commit hook. **For Epic 8 retro consideration:** investigate whether a `pre-commit` hook on `AC-` token diffs is feasible.

### 5. Story 7.2 closed with 0 net HIGHs — **but the maturity-signal thesis is confounded**

Stories 7.1, 7.3, 7.4 each closed with at least one HIGH finding applied. Story 7.2 closed with **0 net HIGHs applied + 5 MEDs applied + 3 LOWs applied** (per sprint-status L114; the original draft of this retro omitted the 3 LOWs from the count). The MEDs were real (e.g., `trials_per_task<1` ValueError validation; untested summary assertions for 4 aggregate stats; docstring drift on always-fail null-name case). The LOWs included one Auditor finding amending an AC-7.2.2 pass_at_k drift to a new catalog entry (C58).

**Confounding factors that prevent a clean "maturity signal" reading** *(added 2026-05-25 per cross-LLM review HIGH-1 [Codex] + LOW-2 [Claude]):*

- Story 7.2 ran a **3-reviewer cycle** (Codex + Claude Blind + Auditor), not the standard 4-reviewer cycle. Edge Case Hunter — historically the highest-yield lane for in-depth behavioral probing on data-structure-heavy code (cohort task summaries, pass_at_k math) — was absent. The reviewer-count delta alone may explain a HIGH→MED downgrade.
- Sprint-status L114 records `Codex HIGH-1` as a raw finding that did not survive triage to a net-applied HIGH. The "0 net HIGHs" picture excludes that raw proposal.
- Story 7.2's **pre-create-story drift check** (D-2 `mcp_coverage` vs `adapter_coverage`; D-3 `competing_skills_picked={}` Phase-1 limitation) **caught usability + concept-disambiguation gaps BEFORE authoring** that in prior epics surfaced AT code-review. Some "missing HIGHs" reflect upstream gate effectiveness, not declining review yield.

**Honest framing:** Story 7.2 closed with 0 net HIGHs + 5 MEDs + 3 LOWs against a 3-reviewer cycle with one raw HIGH proposal not surviving triage. N=1 from a confounded sample is not enough to declare a maturity-signal trend. **Track reviewer-normalized raw + applied HIGH/MED/LOW counts across Epic 8 before inferring ROI drift.**

### 6. Story 7.4 caller-gap (DF-7.4-S2 / C61) is the third caller-count-check catch

The `feedback_caller_count_check` norm (Epic 5 retro #6) fires at story-close. Story 7.4 invoked it and confirmed: 6 path constants (`SKILL_PATH_RF_BROWSER`, etc.) + 2 directory constants in `agentskills_discoverability.py` have ZERO callers from the robot test suite (RF uses `${CURDIR}/skills/...` string interpolation; `Library` imports expose keywords, not module-level constants, as RF variables).

The dev catalogued C61 at story-close BEFORE review (correctly applying the norm). Auditor + Codex 2-way LOW confirmed at review. The norm worked as designed.

**Observation, not lesson:** caller-gaps now have a 3-instance track record (Story 5.1 span helpers → 5 stories of latency; Story 6.1 metrics helpers → caught at story-close; Story 7.4 path constants → caught at story-close). The norm moves these from "5 stories of detection latency" to "same-story catch." Reaffirmed load-bearing.
